4 Key Concepts
pom.xml
What: Project Object Model — XML config declaring name, version, Java level, dependencies, plugins.
Analogy: Recipe card listing ingredients + cooking method.
Why: Maven reads it to know what to download and how to build.
Build lifecycle
What: Fixed sequence of phases. Running any phase auto-runs all previous ones.
Analogy: Car assembly line — engine install requires chassis first.
Why: Guarantees repeatable, nothing-skipped builds.
Local .m2 repository
What: C:\Users\name\.m2\repository — cached downloaded jars.
Analogy: Personal pantry stocked once from the store.
Why: Spring Boot pulls 40MB+ of deps; downloads once, reuses forever.
target/ folder
What: Maven's output directory — compiled .class files, test reports, final .jar.
Analogy: Kitchen counter where finished dishes come out.
Why: The .jar here is what gets dockerised and deployed to EC2.
How Maven fits in the DevOps pipeline
GitHub push ──► Jenkins picks up ──► mvn clean package ──► shopease.jar ──► Docker build ──► Deploy to K8s

Maven's job ends when the JAR is produced. Docker picks it up from there. You'll wire this together in Jenkins on Day 10.

Best practices vs mistakes
✅ Do This ❌ Never Do This
Use mvn clean package in all CI pipelines Run mvn package without clean — stale classes cause phantom bugs
Add -DskipTests only temporarily during debugging Skip tests permanently — they catch bugs before production
Commit pom.xml with every new dependency added Commit the target/ folder — it's 40MB+ of compiled junk
Run mvn dependency:tree when debugging classpath issues Add dependencies blindly without checking transitive conflicts
Use fixed versions in pom.xml for production (1.0.0) Ship -SNAPSHOT versions to production — they are mutable
Check target/surefire-reports/ for detailed test failure info Ignore test failures thinking they're flaky — investigate every one

🟢 Beginner Questions
BEGINNER
Q: What is Maven?
A: Build automation tool for Java. It manages dependencies, compiles code, runs tests, and packages to a .jar. Configured via pom.xml.
BEGINNER
Q: What is pom.xml?
A: Project Object Model — declares project name, version, Java level, dependencies, and plugins. Maven reads it to know what to download and how to build.
BEGINNER
Q: What command creates a JAR?
A: mvn clean package. Clean deletes old artifacts; package compiles, tests, and creates the .jar in target/.
BEGINNER
Q: What is the target/ folder?
A: Maven's output folder — .class files, test reports, and final .jar. Never commit to Git. Always in .gitignore.
🟡 Intermediate Questions
INTERMEDIATE
Q: Difference between mvn package and mvn install?
A: package creates the JAR in target/. install does that AND copies it into the local .m2 repository so other local projects can depend on it.
INTERMEDIATE
Q: What is a SNAPSHOT version?
A: A version ending in -SNAPSHOT (e.g., 1.0.0-SNAPSHOT) is still in development and can change. Maven re-checks for updates. Remove it for stable releases.
INTERMEDIATE
Q: What does -DskipTests do and when is it OK?
A: Skips test execution. Acceptable only temporarily during development iteration. NEVER use in CI/CD pipelines — tests are the safety net.
INTERMEDIATE
Q: How does transitive dependency resolution work?
A: If your pom.xml depends on Library A, and A depends on B, Maven downloads B automatically. Use mvn dependency:tree to see the full chain.
🔴 Scenario & 🎯 Trick Questions
SCENARIO
mvn test fails with OutOfMemoryError on CI server (512MB RAM). Fix?
A: Set MAVEN_OPTS in Jenkinsfile: environment { MAVEN_OPTS = '-Xmx512m -Xms256m' }. Or set globally on Jenkins server as env var.
SCENARIO
Code compiles on dev machine but fails in Jenkins. Maven cause?
A: Missing dep in pom.xml but exists in dev's .m2 cache. Jenkins clean env reveals it. Fix: run mvn dependency:resolve first, use mvn clean package.
TRICK
Does mvn package also run tests?
A: YES — this trips everyone. package runs validate+compile+test+package in order. To skip tests you must explicitly add -DskipTests.
TRICK
What's the difference between <dependency> and <plugin>?
A: dependency is a library bundled into your JAR (Spring Boot, H2). plugin is a build tool (Surefire, SonarQube scanner) — only used at build time, never in your JAR.
