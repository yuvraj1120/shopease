💡 4 Key Concepts of Day 2
🌿 Branch
A parallel copy of your code. Work on it safely — main stays untouched. Like photocopying a doc before editing.
# create + switch in one command
git checkout -b feature/name
Switched to new branch 'feature/name'
🔀 Pull Request (PR)
Formal request to merge your branch into main. Opens review discussion. No professional team merges without a PR.
# after push → go to GitHub
🟡 "Compare & pull request"
→ fill title + desc → Create PR
🔀 Merge Strategies
3 ways Git can combine branches: Merge (preserves history), Squash (1 clean commit), Rebase (linear, dangerous on shared).
# on GitHub PR page choose:
Merge → keeps all commits
Squash → 1 commit on main
Rebase → linear (use with care)
🔐 Branch Protection
GitHub rules that PREVENT direct pushes to main. Forces PRs. Blocks force-push. Every real company has this.
# GitHub → Settings → Rules
→ New ruleset → protect-main
→ Require PR + Block force push

🔀 3 Merge Strategies — Know the Difference
🔀 Merge Commit ✅ Use this
Combines branches + creates merge commit M. Full history visible. Default on GitHub.
main: C1─C2──────M \ / feat: F1─F2─F3
🗜️ Squash Merge Cleanup
F1+F2+F3 become ONE commit on main. Good for messy "oops", "fix" commits.
main: C1─C2─[F1+F2+F3] ↑ 1 clean commit no branch visible
♻️ Rebase ❌ Shared
Replays commits on top of main. Linear history. NEVER on shared branches.
main: C1─C2─F1'─F2'─F3' ↑ linear, no merge commit rewrites commit hashes!

🟢 Beginner Questions (1–2 yr level)
1
What is a Git branch and why do we use it?
A branch is an independent line of development. Work on new features without touching main. When done + reviewed, merge it back. Without branches, developers overwrite each other.
git checkout -b feature/name
2
What is a Pull Request?
A PR is a proposal to merge one branch into another. It opens a review discussion, can trigger CI/CD pipelines, and merges only after approval. No professional team merges without a PR — it's a gate for code quality.
3
What does git pull do?
git pull = git fetch + git merge combined. It downloads changes from remote AND applies them to your current branch. Always pull before starting new work to avoid conflicts.
git pull origin main
4
Why delete branches after merging?
Stale merged branches cause confusion — devs don't know which are active vs done. Most teams auto-delete on merge. Keeps the repo clean and easy to navigate. Your shopease uses this practice.
🟡 Intermediate Questions (3–5 yr level — your target)
5
Merge vs Squash vs Rebase — when to use each?
Merge — keeps full history + merge commit. Safe for all branches.
Squash — combines all commits into 1 clean commit. Use for messy feature branches.
Rebase — linear history, no merge commit. NEVER on shared branches — rewrites hashes.
6
What is branch protection and why is it critical?
Branch protection prevents direct pushes to main, requires PRs, and blocks force-push. Without it one git push --force can destroy production history. We set this up today as a GitHub Ruleset named "protect-main".
7
What is git stash and when do you use it?
git stash temporarily shelves uncommitted changes so your working directory is clean. Use when you're mid-feature and need to urgently fix a bug on another branch. git stash pop restores your work exactly.
8
git fetch vs git pull — what's the difference?
git fetch — downloads changes from remote but does NOT apply them. You inspect first.
git pull — fetch + merge in one step. Best practice: fetch first, then merge manually in critical situations.
🔴 Scenario + 🎯 Trick Questions (FAANG / MNC level)
9
You pushed AWS credentials to GitHub. What do you do RIGHT NOW?
1. AWS Console → IAM → delete the key immediately (bots steal in minutes)
2. Generate new key pair
3. Remove file locally + add to .gitignore
4. git rm --cached file.pem → commit → force push
5. Use BFG Repo Cleaner to scrub all history
6. Report to security team
→ Prevention: .gitignore set up on Day 2!
10
Can you recover a deleted branch?
YES — this trips most people. Run git reflog to find the last commit hash of the deleted branch. Then: git checkout -b recovered-branch <hash>. Git keeps reflog for ~30 days. Never panic — check reflog first!
11
TRICK: What's the difference between git reset and git revert?
git reset — moves branch pointer back, REMOVES commits from history. Dangerous on shared branches — never use on pushed commits.
git revert — creates a NEW commit that undoes changes. Safe on main. Always use revert on production branches.
12
TRICK: Your branch is 3 weeks old, main moved 50 commits. What do you do?
Two options: (1) git merge main into your branch — safe, full context preserved. (2) git rebase main — linear history but rewrites your commits. For 3 week old branch always choose merge — safer, easier conflict resolution with full context.
